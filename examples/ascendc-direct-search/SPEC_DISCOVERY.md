# AscendC Direct Invoke Spec Discovery

This is the canonical workflow for an AscendC Direct Invoke goal. It is used by
the Goal Plus main agent after triage and before Search Mode. It does not define
a new Agent workflow or a runtime plugin.

## Boundaries

- The user supplies an operator goal, approximate shape/dtype requirements, and
  reference hints. The user does not prepare a task directory or verifier.
- The main agent owns task normalization, reference resolution, verifier
  generation, readiness, freezing, and Search orchestration.
- Candidate workers own only the allowed AscendC implementation files. They may
  choose any implementation and optimization strategy within that edit surface.
- Generated AKG and CANNBot files are implementation knowledge only. They
  cannot define the Oracle, cases, score, edit surface, or workflow. The frozen
  task and resolved reference contracts take precedence over source-specific
  project conventions in those files.
- Do not load or invoke an external AscendC Agent, plugin, orchestration skill,
  or operator-development workflow. This document plus the frozen task-local
  `_skills/` are the complete AscendC instructions for this scenario.
- Goal Plus Runtime remains domain-neutral and sees only a source path, frozen
  artifacts, verifier commands, pass/fail, and a finite numeric metric.

## 1. Normalize The Request

Write `_task/operator_request.json` in a new source-owned workspace. Load
`examples/ascendc-direct-search/request.schema.json` and use a standards-based
JSON Schema validator to validate the file as an instance of that schema; a
manual field checklist or JSON parse alone is not validation. This file is
generated by the main agent from the raw goal; it is not another input the user
must author.

Resolve approximate shapes into explicit constraints and representative
examples. Do not invent semantics that conflict with a named reference. Keep an
open question only when semantics, output structure, or acceptable error cannot
be determined from the goal and references.

Every request must resolve these fields before readiness:

- registered operator name and Direct Invoke mode;
- ordered inputs and attributes;
- output count, shape relations, and dtype relations;
- supported concrete dtypes;
- reference roles and exact provenance;
- correctness and performance case policy;
- target NPU identity;
- metric and promotion rule.

## 2. Resolve Reference Hints

References have explicit roles:

| Role | Meaning |
|---|---|
| `semantics` | Defines the mathematical/API behavior. |
| `golden` | Supplies or supports an executable correctness Oracle. |
| `cases` | Supplies shapes, dtypes, attributes, ranges, or weights. |
| `tolerances` | Supplies accepted numerical error policy. |
| `baseline` | Supplies a measurable non-candidate implementation. |
| `scoring` | Supplies an executable metric, direction, aggregation, weights, and acceptance thresholds. |
| `implementation` | Supplies implementation evidence and context; it does not define the verification contract. |

Supported evidence kinds include `cannbench`, `akg`, `pytorch`, `local`, and
`documentation`. The main agent may inspect or clone a named public source when
it is absent locally. Record the resolved repository URL, exact commit, selected
paths, SHA-256 hashes, roles, and any transformations in
`_task/reference_manifest.json`.

Provider rules:

- CANNBench: use `proto.yaml`, `golden.py`, cases, explicit thresholds, and its
  evaluator/scoring implementation only for the roles those files actually
  support. When the user requests CANNBench validation, treat its scoring
  contract as a `scoring` reference. Full reference acceptance means all
  selected official cases pass precision and the ranking metric is the finite
  official report `overall_score`, maximized. Run the performance-enabled
  evaluator; `--no-perf`, a precision margin, pass rate, or raw latency is not a
  substitute ranking metric. Do not invoke its Agent pipeline.
- AKG: treat kernels, tests, API declarations, and examples as evidence. If no
  independent executable Golden exists, generate a PyTorch/NumPy Golden from
  the resolved semantics and self-test it; do not use the candidate as Oracle.
- PyTorch: pin the callable and environment used for the Golden. Expand aliases
  and default attributes into the normalized contract.
- Local files: hash every selected file and reject paths that disappear or
  change before freezing.
- Documentation: it can clarify semantics but cannot by itself prove an
  executable correctness gate or baseline.

When references disagree, prefer the user's explicit API contract, then an
executable task definition, then tests, then prose. Record the conflict and do
not enter Search until the resolved contract is coherent.

## 3. Materialize The Workspace

Create a fresh workspace outside `.gp/`, `.search/`, and ignored scratch roots.
Use `template/` only as a source template. Rename and adapt its operator, kernel,
host, extension, registration, Meta, CMake, SoC, and host-target details to the
normalized request. Do not leave `add_custom` placeholders or template test
scripts as authoritative verification.

Resolve the local AKG Git checkout from `AKG_ASCENDC_SKILLS_ROOT`, `AKG_ROOT`,
or a known sibling path. The configured path may be the AKG repository root or
the nested `akg_agents/python/akg_agents/op/resources/skills/ascendc` directory;
the materializer normalizes it to the Git root. Resolve CANNBot from
`CANNBOT_SKILLS_ROOT` only for the declared supplements. If a required checkout
is absent, clone the repository declared in `knowledge.sources.json` into a
source-owned dependency location.

Run `materialize_knowledge.py` with exact full Git commits and write directly
to the new workspace's `_skills/`. The materializer reads Git objects rather
than checkout working trees, expands the curated AKG `SKILL.md` and reference
tree, applies the explicit CANNBot supplement allowlist and fail-closed
sanitization policy, copies both source licenses, and records resolved commits,
materializer hash, source blobs, hashes, and transformations in
`_skills/manifest.json`. Use the commits pinned by `knowledge.sources.json`
unless the raw goal explicitly requires different fully audited commits. Do
not copy or mount either checked-out Skill directory directly.

The internal materialization step is:

```bash
python examples/ascendc-direct-search/materialize_knowledge.py \
  --akg-root <akg-checkout-or-ascendc-skill-directory> \
  --cannbot-skills-root <cannbot-skills-checkout> \
  --source-ref akg=<full-40-character-git-commit> \
  --source-ref cannbot=<full-40-character-git-commit> \
  --output-dir <generated-workspace>/_skills
```

This is a Main-Agent Spec Discovery command, not a user preparation step.

Generate `_task/TASK.md` that tells candidates to read the normalized request,
reference manifest, target platform, and generated `_skills/README.md`.

The first implementation profile is intentionally narrow:

- one Tensor output;
- first schema argument is a non-optional Tensor;
- output shape and dtype are derived unambiguously from the request;
- Direct Invoke through a PyTorch NPU extension;
- no nested Candidate Agent or source-repository workflow.

Reject unsupported requests during Spec Discovery instead of silently forcing
them into this profile. Later broadcast, reduction, matmul, dynamic-output, and
multi-output profiles may use the same normalized request format with different
scaffolds.

## 4. Generate The Verifier Bundle

Generate task-specific files under `_oracle/`, `_task/`, and `_verifier/`.
There is no checked-in universal AscendC verifier to copy. Reuse trustworthy
public libraries and reference tools where practical, but freeze every command
wrapper and configuration file that determines the result.

Required outputs:

```text
_task/operator_request.json
_task/reference_manifest.json
_task/target_platform.json
_task/search_policy.json
_task/baseline.json
_task/verifier_readiness.json
_oracle/reference.py
_oracle/cases.jsonl
_oracle/tolerances.json
_verifier/<generated build/precision/benchmark/promotion entrypoints>
_skills/
operator/
```

The verifier may call a pinned CANNBench, AKG, or local tool, but the wrapper
must turn its output into Goal Plus' process contract: zero exit only on a valid
candidate and a final JSON object with the finite ranking metric.

### Read-Only Invocation Contract

Treat the Candidate workspace passed as the verifier working directory as an
immutable input. Every process and promotion invocation must require the unique
`GOAL_PLUS_VERIFIER_TMPDIR` supplied by Goal Plus and create a fresh
`$GOAL_PLUS_VERIFIER_TMPDIR/workspace` copy. Run the build, installation,
precision runner, official benchmark/profiler, and report parser only from that
writable copy. Redirect all generated state into the invocation directory,
including, without exception:

- `build/`, CMake/Ninja state, compiler caches, and packaging state;
- `dist/*.whl`, installed wheel staging, and generated extension `.so` files;
- CANNBench or other official-runner JSON/profiler reports;
- Search or Promotion precision evidence and final result files.

Use the runtime-provided
`goal_plus.verifier_support.isolated_verifier_workspace` context manager to
materialize this copy instead of maintaining another ad hoc copy/ignore
implementation in each generated verifier.

Use the object returned by the context manager exactly as an object. Never put
the `IsolatedVerifierWorkspace` representation itself into a shell command.
For a CANNBench submission, require every build/package input and launch the
evaluator from `temp_root`, which is neutral and cannot shadow the installed
wheel with the live source package:

```python
required = (
    "build.sh",
    "setup.py",
    "CMakeLists.txt",
    "scripts/build_wheel.sh",
)
with isolated_verifier_workspace(source, required_paths=required) as isolated:
    workspace = isolated.workspace
    temp_root = isolated.temp_root
    reports = temp_root / "reports"
    evaluator_env = sanitized_evaluator_environment(
        python_paths=(pinned_cannbench_src,),
    )
    command = [
        sys.executable,
        "-m",
        "kernel_eval.cli",
        "eval",
        "--source-dir",
        str(workspace),
        "--reports-dir",
        str(reports),
        # Add the frozen task, device, and measurement arguments here.
    ]
    completed = subprocess.run(
        command,
        cwd=temp_root,
        env=evaluator_env,
        check=False,
    )
```

Goal Plus intentionally prepends the live verifier workspace to the verifier
process `PYTHONPATH` so local verifier modules are importable. Do not propagate
that inherited value to an evaluator which installs and imports a Candidate
wheel. Use `goal_plus.verifier_support.sanitized_evaluator_environment` to
rebuild the evaluator child's `PYTHONPATH` from only its pinned framework
sources and other explicit non-Candidate dependencies. A neutral `cwd` alone
is insufficient: an inherited live-workspace `PYTHONPATH` can still shadow the
installed wheel.

Preserve source assets whose names contain `build`: `build.sh` and declared
`scripts/build_run.sh`/`scripts/build_wheel.sh` are inputs, while generated
`build/`, `dist/`, wheels, and extension `.so` files are outputs. Do not use a
glob such as `build*` to filter the copy. Prefer the shared isolation helper,
which excludes untracked generated artifacts without dropping source scripts.

Do not create, modify, or delete any path in the Candidate input, including
ignored paths and `_verifier/`. Do not copy generated state back after the
command. `expected_outputs` must not name ephemeral files under the invocation
directory. Freeze preflight must run the complete commands and reject any
Candidate-workspace side effect before Search starts.

The verifier may retain detailed files inside the invocation directory while
it runs, but those files are temporary. On success, print the complete
structured evidence to stdout and make the last JSON object contain the finite
ranking metric. On failure, print the complete structured failure report to
stdout, including the phase, failed case IDs, relevant runner/build errors, and
evidence mismatch details, rather than printing only a path to a temporary
report; then exit nonzero. Never include credentials or unredacted environment
values in either form of output.
When `GOAL_PLUS_VERIFIER_DIAGNOSTICS_DIR` is present, also copy compact official
runner JSON and failure evidence there. Goal Plus persists that directory under
the run logs after the invocation scratch directory is removed.

### Correctness Invariants

- Shape, output count, dtype, device residency, NaN positions, and Inf positions
  and signs are checked explicitly.
- Finite floating values use explicit per-dtype tolerances.
- Integer and boolean outputs are exact unless the normalized request says
  otherwise and the reference justifies it.
- Every performance case is also a passed precision case during Search.
- A successful precision report records `passed_case_ids`, `cases_sha256`, and
  the hash of the exact built candidate artifact.
- Benchmarking recomputes those values and refuses stale, incomplete, or
  mismatched precision evidence. It must prove that its exact Performance Case
  ID set is a subset of `passed_case_ids`, that `cases_sha256` binds the
  canonical definitions of those cases, and that `artifact_hash` identifies
  the same temporary-workspace binary being benchmarked.

### Case Generation

Start with cases from the selected reference, then reconcile them with the user
constraints. Add missing boundary coverage rather than replacing reference
cases silently. For every supported dtype include, where applicable:

- smallest valid shape;
- representative normal shape;
- aligned and non-aligned element counts;
- tail-block shape;
- largest bounded performance shape;
- attribute boundaries and special numerical ranges.

Use stable case IDs and record the origin of each case. Generate one acceptance
case policy shared by Search and Promotion; do not generate separate Search and
Promotion case lists. Every performance case is a member of the shared
correctness case set. Search precision and Promotion precision use the same
case IDs, Oracle, tolerances, shape/dtype rules, and pass thresholds. Search and
Promotion performance use the same case IDs, weights, baseline/scoring
metadata, measurement protocol, aggregation, metric, and direction.

When a selected reference provides a public acceptance case set and the user
does not explicitly request a smoke subset, the shared correctness policy must
include the complete reference case set. A user-requested subset must be
recorded and reported as a smoke or partial result, never as full reference
acceptance. Add missing user-required boundary coverage to both phases rather
than only to Promotion. A selected Candidate must never first discover a
Candidate-repairable correctness condition during Promotion.

### Verifier Self-Test

Before measuring a baseline, prove that the generated checker accepts the
Golden against itself and rejects independent negative controls for:

- wrong output count;
- wrong shape;
- wrong dtype;
- ordinary finite values beyond tolerance;
- integer mismatch when applicable;
- NaN-position mismatch;
- Inf-position and Inf-sign mismatch.

Write each result as a named boolean in `_task/verifier_readiness.json`, bind it
to the cases, tolerances, reference, generated verifier, and platform hashes,
and require all booleans at freeze preflight.

## 5. Measure And Freeze

Load the target NPU environment before every environment-sensitive command.
Deployment-specific paths come from environment variables and must never be
embedded in a generated task, prompt, SearchSpec, or report:

```bash
source "${GOAL_PLUS_NPU_CONDA_SH:?set GOAL_PLUS_NPU_CONDA_SH}"
source "${GOAL_PLUS_NPU_ENV_SH:?set GOAL_PLUS_NPU_ENV_SH}"
```

Discover and freeze SoC, device ID, CANN, host architecture, torch, and
torch_npu facts. Run the non-candidate reference over all performance cases and
record per-case latency, weights, synchronization policy, warmup/repeat counts,
noise limits, attempts, aggregation, and hashes in `_task/baseline.json`.

Freeze one comparison basis for the whole run. Reference-provided or locally
measured baseline/scoring inputs are both valid, but Search, Promotion, and
every Candidate must use the same frozen inputs and aggregation.

Resolve the ranking contract in this precedence order:

1. an explicit metric and direction in the user goal;
2. the executable scoring contract of the selected `scoring` reference;
3. the default contract below.

When a selected reference defines scoring, adapt its score into the finite Goal
Plus metric and apply the same frozen scorer and comparison inputs to the
Baseline and every Candidate. Record the source and transformation in the
reference manifest and search policy.

For CANNBench, the SearchSpec metric is `cannbench_score`, direction `maximize`,
and its value is the official performance-enabled report `overall_score` after
all official precision cases pass. Parse it with
`goal_plus.verifier_support.parse_cannbench_report`; do not add `--no-perf` and
do not rank with `precision_margin`. A missing, non-finite, partial-case, or
precision-failing report exits nonzero and emits no ranking metric.

When neither the user nor a selected reference defines scoring, require every
shared correctness case to pass and rank valid Candidates by minimum measured
time. The default ranking contract is:

```json
{
  "metric_name": "weighted_latency_us",
  "metric_direction": "minimize"
}
```

The process verifier must perform build, shared-contract precision, and
benchmark in one bounded command and emit the metric only after all hard gates
pass. The promotion verifier independently repeats the same frozen acceptance
contract from a clean build and fresh measurements. Promotion may collect
additional non-gating diagnostics, but it must not add a case, tolerance,
anti-cheat rule, profiler requirement, threshold, or other rejection condition
that Search did not enforce. Any required formal profiler or anti-cheat check
is part of both phases.

All SearchSpec verifier commands that use the target NPU, including every
process and promotion command, must declare the same non-empty resource lock,
for example `"resource_lock": "ascend-npu:<device-id>"`. Baseline measurement
must not overlap a locked verifier invocation. Candidate workers may still run
with `max_parallel > 1`, but the shared NPU benchmark/profiler region is
serialized so candidates cannot corrupt reports or distort each other's
latency. A different lock name between Search and Promotion is a readiness
failure.

Save a high-confidence `goal_plus_save_spec_draft` only after the generated
ranking command succeeds on the source workspace with a finite metric and all
open questions are resolved. Immediately before `search_freeze_spec`, validate
the final `_task/operator_request.json` again against the canonical
`examples/ascendc-direct-search/request.schema.json` with the JSON Schema
validator and stop on any validation error. Freeze the generated verifier,
Oracle, cases, policies, manifests, template-derived fixed files, and generated
`_skills/` bundle before creating the Search run.

The source workspace is the Seed for first-generation Candidates. Before
freezing, the main agent must provide a minimally correct Seed that builds and
passes every shared correctness case; it does not need to be performant.
Candidate workers start from that verifier-backed Seed and own subsequent
implementation and optimization attempts. The non-Candidate Baseline is a
separate comparison implementation or data record, not the Seed.

## 6. Search And Promotion

Use the standard Goal Plus/Search tools. Candidate prompts contain the
normalized task, edit surface, delivery format, knowledge location, verifier
entrypoint, and the resolved `implementation` references relevant to the task.
Candidate generation remains unconstrained within the allowed files.

Selection and promotion use verifier-backed Git revisions. Promotion Patch
content must be generated from the immutable selected commit, never from a live
Candidate workspace. A live workspace mutation after promotion evidence must
either be rejected or be unable to affect the Patch.

The final raw-goal audit checks the original operator API, requested shape/dtype
coverage, selected report, promotion evidence, and Git-applicable Patch. Passing
only the generated SearchSpec is insufficient when the raw goal requested
additional integration work.
